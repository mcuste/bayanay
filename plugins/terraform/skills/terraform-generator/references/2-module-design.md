# §2 — Module Design

## When to Use

- Resource group is reused across 2+ configurations
- Abstraction has a clear, stable interface
- Need to enforce organizational standards (naming, tagging, encryption defaults)
- Resource group has independent lifecycle concerns

## How It Works

Modules are Terraform's primary abstraction mechanism. Organize into three tiers:

| Level  | Name                   | Description                                                          | Example                                |
|--------|------------------------|----------------------------------------------------------------------|----------------------------------------|
| **L1** | Resource modules       | Thin wrappers around 1-3 related resources. Expose most arguments.   | `terraform-aws-s3-bucket`              |
| **L2** | Infrastructure modules | Compose L1 modules into a functional unit. Encode org opinions.      | `terraform-aws-eks-cluster`            |
| **L3** | Composition modules    | Wire L2 modules into a complete environment. Minimal config surface. | `terraform-aws-production-environment` |

**Composition patterns:**

- **Dependency inversion (preferred):** Modules accept dependencies as inputs, don't create them. Makes modules composable and independently testable.
- **Data-only modules:** Modules containing only data sources for querying existing infrastructure. Encapsulates lookup logic.
- **Multi-cloud abstraction:** Common variable contract with provider-specific implementations.

**When NOT to create a module:**

- One-off configuration specific to this deployment
- Adding indirection without reuse benefit
- Simple, flat resource graph (< 5 resources)
- Passing more variables into a module than the module has resources

**Standard module structure:**

```
modules/vpc/
  main.tf          # Resource definitions
  variables.tf     # Input variable declarations
  outputs.tf       # Output value declarations
  versions.tf      # required_providers and terraform version constraints
  locals.tf        # Local value computations (optional)
  data.tf          # Data source lookups (optional)
  README.md        # Documentation (auto-generated by terraform-docs)
  tests/           # Native test files (TF 1.6+)
  examples/        # Usage examples (recommended for shared modules)
```

## Code Snippet

```hcl
# Dependency inversion — module accepts VPC, doesn't create it
module "network" {
  source = "./modules/network"
  cidr   = "10.0.0.0/16"
}

module "cluster" {
  source     = "./modules/eks-cluster"
  vpc_id     = module.network.vpc_id
  subnet_ids = module.network.subnet_ids
}

# Module input/output contract
# variables.tf
variable "vpc_cidr" {
  description = "CIDR block for the VPC"
  type        = string

  validation {
    condition     = can(cidrhost(var.vpc_cidr, 0))
    error_message = "Must be a valid CIDR block."
  }
}

variable "environment" {
  description = "Deployment environment"
  type        = string

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Environment must be dev, staging, or prod."
  }
}

# outputs.tf
output "vpc_id" {
  description = "ID of the created VPC"
  value       = aws_vpc.main.id
}

output "private_subnet_ids" {
  description = "IDs of private subnets"
  value       = aws_subnet.private[*].id
}

# Version pinning
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"    # allows 5.x, blocks 6.0
}

module "custom" {
  source = "git::https://github.com/myorg/terraform-modules.git//vpc?ref=v2.3.1"
}
```
